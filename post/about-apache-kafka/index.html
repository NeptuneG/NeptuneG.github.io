Kafka一瞥

Kafka的初衷

要理解Kafka的设计，如果直接研究Kafka在LinkedIn业务中的必要性，倒不如先看一篇IEEE的论文“Building LinkedIn’s Real-time Activity Data Pipeline”。简单来说是这样的。

LinkedIn需要处理两大块数据。一个是由大量用户在网站上产生的 Activity Data 。这部分数据通过Hadoop的批处理进行机器学习后可以用户改善内容推荐等服务。另外一部分则是 系统日志 。这些数据被实时处理后被一些服务所监视。这些都最近一些网站常见的搞法。

然而， Activity Data 可能因为经过Hadoop批处理之后的的特化而不能再被实时处理，从而也就造成监视服务的延迟。对于 系统日志 而言，经过长时间的实时处理的特化，这些数据将不能再被容量规划和系统调试使用。为了改善这种情况，有必要以最小的成本来统一各个不同类型的数据。而且像LinkedIn这样以数据为业务核心的公司，有必要让数据能够被不同的团队简单的使用。

为了解决这些问题，开发一个从不同系统收集大规模不同类型数据并使其统一的平台的过程开始了。

这个过程最开始是在一些既有的消息系统（如论文中提到的ActiveMQ）的基础上进行再开发。但是在处理产品级数据是会有以下问题发生。

  • 并行地读取消息队列中消息的时候，有必要记录每条消息被谁读走。因此大量的数据被使用后会导致内存不足，从而对性能造成严重的影响。
  • 批处理和实时处理同时读队列的时候，因为需要将数据进行拷贝，从而使得系统的效率降低。

这些问题促使了 Kafka 的产生。 Kafka 着眼于以下几点。

  • 各类数据的统一使用
  • 各系统读同一份数据
  • 并行读取数据，高吞吐数据处理

Kafka的原理

Kafka由 Broker ， Producer 和 Consumer 三部分组成。Producer 负责发布消息，Consumer 读取消息。而 Broker 作为Kafka的核心，构成群集维护在 Consumer 和 Producer 之间收发消息的消息队列。

消息的交互

Kafka通过 Topic 来进行消息的交互。 Topic 类似于消息的类别。如，可以将和搜索相关的数据以Search的 Topic 名发送至 Broker 。在有此类数据的需求时，就通过Search这个 Topic 来向 Broker 请求数据。

Pull vs Push

是 Broker 将数据Push到 Consumer 还是 Consumer 从 Broker 把数据Pull过来？这对整个消息系统的设计会产生很大的影响。当然各方都有支持和反对的声音。但是 Kafka 还是选择使用Pull型的 Consumer 。如果以Push的方式，对于处理各式各样不同的 Consumer 来说会比较困难。而且 Broker 必须对数据流量等信息有所感知。 Kafka 立意于以最快的速度消费数据，如果因为一些预想不到的原因，如网络堵塞等，误判了数据的发送量，则有可能带垮 Consumer 。如果以Pull的方式， Consumer 可以自行进行管理以适当的速率对消息进行消费。
